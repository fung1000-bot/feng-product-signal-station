# 枫的产品信号站 - 2026-07-17

## 今日主判断

国内 AI 编程用户正在从“模型哪个好”转向“现在能不能买到、值不值得用、能否稳定完成任务”；今天更适合做一次真实任务测评和内容验证，而不是立刻复刻一个完整 SaaS。

## 今日最值得做的一个方向

K3 Coding Plan 实测卡片 / 面向国内 AI 编程学习者和个人开发者 / 为什么今天：一条关于 Kimi Coding Plan 与 K3 的讨论获得 154,901 次曝光、429 个赞和 207 条回复，且讨论集中在可用性和供应不确定性 / 与枫的连接：可以同时产出一篇“国内 Coding Plan 怎么选”的内容和一个极简测评工具雏形 / 2 小时能做什么：选 3 个真实编码任务，记录 K3 的完成率、耗时、返工次数、上下文处理和费用，整理成一张可分享的对比卡片 / 周末能扩成什么：做成“模型 × Coding Plan × 任务类型”的公开测评页，增加用户提交结果和历史版本记录。

## 今日 Top 3 信号

### 信号 1

🔍 信号：关于“先购买 Kimi Coding Plan，再使用 K3”的帖子获得 154,901 次曝光、429 个赞、207 条回复、12 次转发和 7 次引用。[原始证据](https://x.com/Khazix0918/status/[REDACTED_CREDIT_CARD])

白话说：用户关心的不是抽象参数，而是“现在能不能用、要不要先买、会不会很快买不到”。

分析：这更像一个强烈的即时决策场景。即使帖文中的供应判断尚未被其他证据验证，讨论量已经说明 Coding Plan 的购买与使用决策有内容需求。适合先做实测内容，不适合直接把“买不到”写成事实。

关键判断：今天最值得验证的是“真实编码任务测评”而不是再做一篇模型参数介绍。可测试标题：“我用 K3 连续做 3 个真实小功能，Coding Plan 到底值不值”。

反向视角：这可能只是一次热点传播，不代表持续付费意愿；如果 K3 很快普及或供应恢复，围绕稀缺性的产品价值会迅速下降。

### 信号 2

🔍 信号：TrustMRR 当前展示了多个可观测收入的 AI 或开发者工具项目：RenderLab 收入 95k、售价 10M；Videotok 收入 8.5k、售价 75k；Body Fat Estimator 收入 335、售价 10k。[RenderLab](https://trustmrr.com/startup/renderlab) · [Videotok](https://trustmrr.com/startup/videotok) · [Body Fat Estimator](https://trustmrr.com/startup/body-fat-estimator)

白话说：小型 AI 产品从低收入到较高收入都有真实交易或出售挂牌，但“有收入”不等于“容易复刻”。

分析：其中 Body Fat Estimator 的挂牌收入较低、售价也相对低，理论上更接近一周内可以做 MVP 的体量；RenderLab 的收入更高，但售价和产品复杂度信息不足，不应直接作为个人复刻目标。现有证据只说明品类和收入挂牌，不足以判断具体功能、获客渠道或留存。

关键判断：TrustMRR 更适合作为“寻找窄场景 AI 工具”的雷达，而不是直接照搬产品。今天可以优先拆 Body Fat Estimator 的输入、输出和付费点，但健康类产品需要把结果定位为估算，不能包装成医疗判断。

反向视角：健康估算涉及准确性、责任和用户信任；如果没有可靠数据或专业边界，可能比看起来更难验证。Videotok 的具体功能也没有在输入证据中说明，不能据此设计产品。

### 信号 3

🔍 信号：Cars24 使用 OpenAI 的语音和聊天 Agent，每月处理超过 1M 次对话分钟，并恢复 12% 的流失线索。[原始证据](https://openai.com/index/cars24)

白话说：AI Agent 的价值开始被用“处理了多少对话、挽回了多少线索”衡量，而不只是演示效果。

分析：这给中文 AI 副业内容提供了一个更具体的切入点：围绕“一个小微商家如何把咨询、跟进和流失召回串起来”做实操，而不是泛泛讨论 Agent。证据来自企业案例，不能直接推断个人开发者也能取得相同效果。

关键判断：可做一个“线索召回 Agent”小实验：用模拟咨询记录，让 Agent 对未成交用户分类、生成跟进话术并安排下一步动作。它比做通用聊天机器人更容易形成可展示的业务闭环。

反向视角：Cars24 的规模、数据和业务流程与个人项目差异很大；没有真实商家线索时，实验只能证明流程可行，不能证明有人愿意付费。

## 机会拆解

痛点：AI 编程用户面对 Coding Plan 和模型更新时，缺少基于真实任务的稳定比较；小微商家面对咨询线索时，也缺少低成本的跟进和召回流程。

明确买家：第一顺位是 AI 编程学习者、个人开发者和小微创业者；第二顺位是有大量咨询但缺少销售跟进能力的小微商家。

已有替代：模型官方宣传、社交平台口碑、通用聊天机器人、人工跟进。当前证据没有证明这些替代方案存在明确缺口，因此需要先做小规模验证。

小产品入口：  
1. “Coding Plan 实测卡片”：任务模板、模型输出、耗时、返工次数、费用和结论。  
2. “线索召回 Agent”：导入一小批脱敏咨询记录，输出客户分层与跟进动作。  
3. “窄场景 AI 工具拆解”：从 TrustMRR 的 AI 产品挂牌中筛选可验证方向，但不直接购买或照搬。

可验证数据：  
- K3 测评：任务完成率、首次可运行率、返工次数、耗时、上下文长度和实际成本。  
- 内容验证：标题点击、收藏、评论中是否出现“想测什么”“怎么买”“能不能替代某工具”等明确需求。  
- Agent 验证：模拟或获得授权的线索中，分类准确率、话术可执行性和人工节省时间。  
- TrustMRR 方向：只记录公开挂牌中的收入、售价和倍数；不把挂牌信息当作产品真实经营全貌。

风险：K3 相关讨论可能是短期热点；Coding Plan 的价格和可用性可能变化；企业 Agent 案例难以外推；健康类 AI 工具存在责任风险；TrustMRR 页面只提供产品名称、类别和挂牌指标，缺少功能、用户和获客细节。

## 反向视角

今天可能不值得直接做一个完整的“模型测评平台”或“线索召回 SaaS”。前者需要持续更新和足够测试样本，后者需要真实业务数据和授权。更稳妥的顺序是：

1. 先用 3 个真实编码任务做一次 K3 测评；
2. 将结果发布成一篇可验证的内容；
3. 根据评论和私信中的具体需求，再决定是否做工具；
4. 只把 TrustMRR 当作候选池，不因收入数字直接判断可复刻性。

## 今天不要追

1. 2.5T/2.8T 模型训练基础设施：证据描述的是数万张 GPU、多机多卡和超大规模训练，明显不适合个人一周 MVP 验证。[相关讨论](https://x.com/mylifcc/status/[REDACTED_CREDIT_CARD])  
2. 1M 上下文和新注意力机制的底层模型研发：虽然讨论提到 KDA、Attention Residuals 等机制，但个人无法据此快速做出可验证产品，且当前证据主要来自同一来源的技术解读。[相关讨论](https://x.com/mylifcc/status/[REDACTED_CREDIT_CARD])
